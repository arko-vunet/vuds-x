# vuSmartMaps Design Principles

**For vuDS-x implementation and all screen/component generation.**

This document defines the design principles, interaction philosophies, and concrete rules that govern how vuSmartMaps looks, feels, and behaves. Every screen — whether hand-built by a developer, designed in Figma, or generated by an AI tool — must conform to these principles. Deviation from any principle requires explicit justification.

vuDS-x (the VuNet Design System) is the enforcement mechanism. These principles are what it enforces.

---

## Context: What vuSmartMaps Is

vuSmartMaps is an observability and business-journey-monitoring platform. Its users — SREs, platform administrators, DevOps engineers, and business stakeholders — spend hours inside it, often under pressure. They monitor production infrastructure, triage alerts at 3am, configure data pipelines that affect real systems, build dashboards that executives read, and analyze logs to diagnose outages.

The product's heritage is a Grafana fork. It has 17+ distinct interaction patterns across dashboards, alert consoles, pipeline builders, log explorers, report builders, RUM session replay, and AI-assisted troubleshooting (Ved). It is transitioning from an on-premises deployment model to multi-tenant SaaS.

These two facts — high-stakes operational use and SaaS transition — shape every principle below.

---

## The Governing Philosophy

**The product must feel like a precision instrument, not a consumer application.**

vuSmartMaps is not trying to delight users with novelty. It is trying to make them effective under pressure. Every design choice optimizes for three qualities, in this order:

1. **Clarity** — Can the user understand what they're looking at in under two seconds?
2. **Speed** — Can they do what they came to do without unnecessary steps?
3. **Trust** — Does the interface give them confidence that what they see is accurate and that their actions are safe?

When these qualities conflict — and they will — resolve in favor of clarity first, then speed, then trust. A clear interface that takes an extra click beats a fast interface that's ambiguous. A speedy workflow that makes users second-guess their actions is worse than one that takes a moment longer but confirms their intent.

## Principle 1: Information Density — Compact but Scannable

### The Principle

Show the maximum useful information per viewport without crossing the threshold where density becomes noise.

### What This Means

Users of data-heavy tools develop scanning patterns. They don't read screens — they scan them for anomalies, changes, and values that match mental thresholds. The UI must support this scanning behavior. Compact does not mean cramped. Scannable does not mean sparse. The target is high information density with clear visual grouping.

### Why This Matters for vuSmartMaps

An SRE monitoring a dashboard during an incident needs to see severity counts, affected services, alert timelines, and metric trends in a single viewport. If they have to scroll to correlate two pieces of information, the interface has failed. Similarly, a table with 15 columns of alert data needs to be readable without horizontal scrolling on a standard 1440px monitor.

The Grafana heritage means users already expect data-dense interfaces. The challenge is making them data-dense AND legible, which Grafana's utilitarian styling doesn't always achieve.

### Rules

1. **Base font size: 14px for data-display contexts (tables, metrics, logs, code), 16px for reading contexts (documentation, empty states, onboarding flows).** The 14px base in data contexts matches what observability users are accustomed to (Grafana, Datadog both use 14px). The 16px base in reading contexts maintains legibility for longer text.

2. **Line height: 1.4 for body text, 1.2 for headings, 1.6 for table rows (to create breathable row separation without excessive whitespace).** Table row height should be 36–40px for default density, 28–32px for compact density.

3. **Spacing rhythm: 4px base unit.** All spacing values must be multiples of 4: 4, 8, 12, 16, 20, 24, 32, 40, 48, 64. No other values. This creates a subconscious rhythm that the eye can predict, which makes scanning faster.

4. **Page gutter: 24px on both sides inside the main content area.** This is deliberately narrower than consumer-app conventions (which often use 32–48px) because every pixel of horizontal space matters for wide tables and dashboards.

5. **Section spacing: 24px between major sections on a page. 16px between sub-sections within a group. 8px between related items within a sub-section.** This three-tier spacing hierarchy creates visual grouping without needing explicit dividers for every boundary.

6. **Maximum content width: no global max-width constraint on data pages (tables, dashboards).** Settings pages and form-heavy pages may use a max-width of 720–900px for readability, but data views should use the full available width. Users with ultrawide monitors bought them for a reason.

7. **Data tables must support column resizing and column visibility toggling.** Not all 15 columns need to show by default. Show the most useful 6–8 columns; let users add more. Persist column preferences per user.

### Anti-Patterns

- Adding padding "to make it breathe" when the user's goal is to see more data. Breathing room in observability tools is wasted space.
- Using large cards with heavy padding around individual metrics when a compact metric row would serve better.
- Enforcing a narrow max-width on table pages because "lines shouldn't be too long." That rule applies to prose, not tabular data.
- Using 16px font for table cells. In a 15-column table, this wastes 15–20% more horizontal space than 14px with no readability gain for short cell values.

## Principle 2: Progressive Disclosure — Layered Complexity

### The Principle

Show the minimum information needed for the current task at each level of interaction. Provide clear, predictable paths to deeper detail.

### What This Means

vuSmartMaps is a deep product. It has pipeline configuration with plugin chains, alert rules with YAML payload transformations, dashboard panels with query editors and field overrides. No single screen can show all of this without overwhelming users. Instead, information is layered: summary → detail → configuration → raw code.

Each layer should be self-sufficient for its audience. A business stakeholder reading a dashboard summary should never need to see query syntax. An SRE configuring an alert should be able to access the YAML without hunting for it.

### Why This Matters for vuSmartMaps

The product serves multiple personas with different depth needs:
- **Business stakeholders** need dashboards and high-level alert summaries.
- **Operations engineers** need alert details, runbooks, and contextual metrics.
- **Platform administrators** need pipeline configuration, RBAC, and data storage management.
- **SREs and developers** need log analytics, query editors, and raw JSON/YAML access.

Progressive disclosure ensures each persona finds their depth level without being blocked by complexity intended for another persona — or being drowned in it.

### Rules

1. **Three-layer disclosure model for entity views:**
   - **Layer 1 — List.** Table row with key identifying columns. User scans and selects.
   - **Layer 2 — Summary.** Drawer or detail page with the entity's full read-only state. Tabs group related information. This is where most users stop.
   - **Layer 3 — Configuration.** Edit view, accessed via explicit "Edit" action from Layer 2. Forms, code editors, advanced settings live here.

2. **Drawers for quick inspection; full pages for deep work.** If the user can accomplish their goal in under 30 seconds (checking an alert's status, viewing a record's details), use a right-side drawer (400–600px wide). If they need to edit, compare, or work with multiple sections, navigate to a full detail page. Never force users into a full page navigation for a read-only glance.

3. **Form-code toggle for every configuration entity that has a structured representation.** Alert rules, pipeline configurations, notification routes, data source settings — all of these have underlying JSON or YAML. Provide a "Visual" / "Code" toggle. Sync changes bidirectionally. The visual form is the default; code is for power users. Neither mode should feel like a second-class citizen.

4. **Expandable sections over tabs when the content is short.** If each tab would contain fewer than 3 form fields or a single small table, use collapsible/expandable sections on a single scrollable page instead. Tabs have a cost: they hide content behind a click and break ctrl+F search. Use tabs only when each tab's content is substantial enough to warrant its own viewport.

5. **Advanced settings behind an explicit "Advanced" expandable section, collapsed by default.** Do not intermix rarely-used settings with common ones. If fewer than 20% of users need a setting, it belongs behind "Advanced."

6. **"Show more" and truncation must indicate what's hidden.** Never truncate to "..." without indicating quantity. Use "Show 12 more" not "Show more." Use "+5" badges on tags that overflow, not just a fade-out.

### Anti-Patterns

- Putting everything on one scrollable page with no grouping. This is not progressive disclosure; it's a wall of content.
- Hiding critical information behind too many clicks. If an SRE needs three clicks to see an alert's severity trend, that's two clicks too many.
- Using modals for complex multi-field editing. Modals should be for focused, single-purpose actions (confirm delete, quick create). Complex editing needs a full page or drawer with scrolling.
- Nesting drawers. If a drawer needs to open another drawer, the first task should have been a page.

## Principle 3: State Communication — Every Element Tells the Truth

### The Principle

Every component that can exist in multiple states must visually communicate its current state at all times, without requiring the user to interact with it to discover that state.

### What This Means

In an observability platform, state IS the primary information. A metric is healthy or degraded. An alert is active or cleared. A pipeline is running or failed. A data source is connected or unreachable. The UI must make these states instantly legible through visual encoding — color, iconography, animation, and spatial treatment.

This extends beyond data states to interface states: a button that is loading, a form that has unsaved changes, a table that is fetching data, a section that failed to load.

### Why This Matters for vuSmartMaps

Users trust the interface to reflect reality. If a dashboard shows green when a service is down (because the data fetch failed silently), the product has lied to the user. If a save button looks normal while an API call is in flight, the user might click away and lose their changes. In a platform that monitors production systems, every ambiguous state is a potential incident.

The vuSmartMaps pattern audit identified 17 patterns. Many involve long-running operations: pipeline processing, bulk imports, report generation, alert evaluation. Users need constant, honest feedback about what's happening.

### Rules

1. **Five canonical UI states for every data-dependent component:**
   - **Loading** — Show a skeleton or spinner. Never show stale data without indicating it might be stale. Skeleton screens are preferred over spinners for layout-stable loading (the user sees where content will appear). Spinners are for indeterminate waits inside already-loaded layouts.
   - **Empty** — Show a purposeful empty state with an explanation and a primary call to action. (See Principle 10.)
   - **Populated** — The normal, data-present state.
   - **Error** — Show what went wrong and how to fix it. Never show a blank area where data should be. (See Principle 10.)
   - **Refreshing / Updating** — If data is already displayed and a background refresh is happening, show a subtle indicator (thin progress bar at the top of the component, or a pulsing dot near the refresh timestamp) without replacing the existing data with a skeleton. The user should not lose their scroll position or visual context during a refresh.

2. **Severity states use a fixed color vocabulary (see Principle 8 for exact values):**
   - Critical → Red
   - Error / High → Orange
   - Warning / Medium → Yellow/Amber
   - Info / Low → Blue
   - Healthy / Success → Green
   - Unknown / Pending → Gray

   These mappings are absolute. No component may use red for a non-critical purpose. No component may use green to mean anything other than success/healthy. This is non-negotiable because users develop color-based muscle memory for triage.

3. **Unsaved changes must be indicated.** Any form, editor, or configuration view that has been modified but not saved must show a visual indicator. Options, in order of preference:
   - A dot or asterisk in the tab/page title (like "Alert Config •").
   - A persistent "Unsaved changes" label near the save button.
   - The save button itself changing from a secondary/ghost style to a primary/filled style when there are pending changes.

4. **Long-running operations must show progress.** Any operation that takes more than 1 second must show a progress indicator. Any operation that takes more than 5 seconds should show a progress bar with either percentage or step indication. Any operation that takes more than 30 seconds should show an estimated time remaining or a descriptive status message ("Processing row 1,204 of 5,000"). If progress cannot be measured, use an indeterminate progress bar with a text description of the current phase.

5. **Real-time data must show its freshness.** Any data that updates periodically (dashboards, alert consoles, metric panels) must display a "Last updated" timestamp or a relative time indicator ("Updated 15s ago"). When auto-refresh is active, show the refresh interval. When a refresh fails, show the failure — do not silently show stale data with a current-looking timestamp.

6. **Connectivity state must be visible.** If the application loses connection to its backend (WebSocket disconnect, API timeout), show a banner at the top of the viewport: "Connection lost. Reconnecting..." with a visible retry countdown or spinner. Do not let users interact with controls that will silently fail.

### Anti-Patterns

- Showing a blank white area while data loads. This is ambiguous: is it loading, empty, or broken?
- Using the same visual treatment for "no data" and "error fetching data." These are different states with different recovery paths.
- Hiding loading states behind optimistic UI updates that might not resolve. Optimistic updates are acceptable for low-risk, fast operations (toggling a checkbox). They are unacceptable for configuration changes that affect production systems.
- Animating state changes so aggressively that the user cannot read the intermediate values (e.g., a counter that animates from 0 to 1,247 over 2 seconds — just show 1,247).

## Principle 4: Spatial Predictability — The Layout Contract

### The Principle

Users must be able to predict where content will appear on any page before they see it. Page anatomy is a contract between the product and its users.

### What This Means

Every page in vuSmartMaps, regardless of module, follows the same spatial template. The sidebar is always on the left. The page title and breadcrumbs are always at the top. The primary action button is always in the same position relative to the page header. Tables always have their filter bar above them. This predictability is not boring — it is a productivity feature. Users who have internalized the layout spend zero cognitive effort on navigation and orientation.

### Why This Matters for vuSmartMaps

The product has 17+ distinct page types across 6+ major modules. If each module invents its own layout conventions, users must re-learn spatial relationships every time they switch contexts. When the alert console puts filters on the left and the report builder puts filters on the top, the user's spatial memory is violated. Multiply this across all modules and the product feels disjointed.

The AppShell meta-pattern from the research (Section 6) establishes the container. These rules establish what goes inside it.

### Rules

1. **The AppShell is invariant.** It consists of:
   - **Left sidebar** (collapsible): Navigation. Always present. 240px expanded, 48px collapsed (icon rail). Contains the product's information architecture as a tree.
   - **Top bar** (fixed): 48px tall. Contains breadcrumbs (left-aligned), global search trigger (center or right), user menu, and notification bell (right-aligned).
   - **Main content area**: Everything below the top bar and to the right of the sidebar. This is where page content renders.

2. **Page anatomy within the main content area is standardized:**
   ```
   ┌──────────────────────────────────────┐
   │ Page Header                          │
   │ [Breadcrumb trail]                   │
   │ Page Title          [Primary Action] │
   │ Optional: description / subtitle     │
   ├──────────────────────────────────────┤
   │ Toolbar / Filter Bar (if applicable) │
   │ Search, filters, view toggles, etc.  │
   ├──────────────────────────────────────┤
   │                                      │
   │ Main Content                         │
   │ (table, form, dashboard grid,        │
   │  detail sections, etc.)              │
   │                                      │
   ├──────────────────────────────────────┤
   │ Pagination or Sticky Action Bar      │
   │ (if applicable)                      │
   └──────────────────────────────────────┘
   ```

3. **The Page Header region is never omitted.** Every page has a title. If the page is a detail view, the title is the entity's name. If it's a list view, it's the collection name ("Alerts," "Dashboards," "Data Sources"). The primary action button ("+ Create," "Edit," "Save") is right-aligned in the header, on the same row as the title.

4. **The Toolbar / Filter Bar sits between the header and the content.** It is not part of the content itself. It is sticky (sticks to the top when scrolling) if the content scrolls. Filters, search, column toggles, view mode switches, and bulk action controls live here.

5. **Drawers always open from the right.** They overlay the main content but do not displace the sidebar. Width: 480px default, resizable up to 720px. The drawer has its own header with a close button and the entity's title.

6. **Modals are centered, overlaid on a dimmed backdrop.** Max-width: 480px for simple confirmations, 640px for creation forms, 960px for complex views (diff viewers, import mapping). Modals must never scroll internally if avoidable — if the content is too long for a modal, it should be a drawer or a page.

7. **Sticky elements:** The AppShell top bar is always sticky. The filter/toolbar bar is sticky within the main content area. The save/cancel action bar on configuration pages is sticky at the bottom of the viewport. Nothing else should be sticky. Over-use of sticky elements creates a "shrinking viewport" effect that reduces usable space.

### Anti-Patterns

- Placing the primary action button in different positions on different pages (sometimes in the header, sometimes at the bottom of a form, sometimes floating).
- Using a left-side drawer. Left-side drawers conflict with the sidebar and create a confusing dual-panel situation.
- Putting navigation controls (tabs, breadcrumbs) inside the scrollable content area so they scroll away.
- Having filter controls in different positions: above the table on one page, in a sidebar on another, inside a dropdown on a third. Pick one position (above the table) and use it everywhere.

### Form Layouts and Field Widths

Form-heavy screens are common in vuSmartMaps (alert configuration, data sources, pipelines, notification routes, RBAC). Their layouts must feel deliberate and consistent, not stretched or arbitrary.

#### Page-Level Form Layout

- **Form context types**
  - **Inline forms**: embedded in tables, drawers, or panels. Optimized for density.
  - **Primary-task forms**: dedicated New/Edit pages and major modals. Optimized for focus and comfort.

- **Primary-task forms: centered within the content area**
  - On desktop (≥1280px), the form lives in a **single main column**, horizontally centered within the main content area (to the right of the sidebar, below the top bar).
  - That column has a **max-width between 720px and 960px** and never exceeds it, regardless of viewport width.
  - The column respects the global **page gutters** (24px on each side at desktop), so fields never touch the edges.
  - On laptop / tablet widths (1024–1279px), the same column can shrink down to **100% of available width**, but the max-width still caps it on larger screens.
  - On narrower viewports (<1024px), the form becomes **edge-to-edge with 16–20px padding**, still a single column.

- **Inline and drawer forms**
  - In drawers and side panels, forms **fill the available panel width** up to the panel’s own max-width (e.g. 480–720px).
  - The same internal rules for labels, fields, and spacing apply; only the container width changes.

#### Field Width Rules

- **No blindly 100% width for every input**
  - Field width should **approximate expected input length**, within a consistent grid.
  - Categories:
    - **Short inputs** (ports, small thresholds, 2–4 char codes): `min(160px, ~30–40% of form width)` or inline with other fields.
    - **Medium inputs** (names, IDs, single-line labels): **50–70% of the form column**, but never beyond **480–560px**.
    - **Long inputs** (URLs, queries, JSON/YAML, long descriptions): **full width of the form column**, respecting the column’s max-width.

- **Grid for multiple fields per row**
  - Use a **12-column grid inside the form column**:
    - Common case: labels/help + fields in a simple vertical stack, but when two fields share a row:
      - Two related fields → 6/6 split at desktop, stacked vertically below **1024px**.
      - Three small fields → 4/4/4 split at desktop, stacked vertically below **1024px**.
  - This gives LLMs a clear pattern: “two related fields in one row → 6/6; otherwise stack”.

- **Consistency**
  - A given **field type in a given context** (for example, “threshold value” on alert config) should always use the same width pattern everywhere it appears.
  - Avoid rows where one field is full-width and the neighbor is tiny; either align them to the grid (e.g. 8/4) or move the small field inline into the label/help text.

#### Labels, Helper Text, and Alignment

- **Label placement**
  - Default for primary-task forms: **labels above fields**, left-aligned with the input (not a separate left column).
  - This works better on narrow viewports and in localized UIs with longer label text.
  - Exception: extremely dense admin-style tables or narrow side panels may use side-by-side labels where necessary.

- **Helper and error text**
  - Always **below the field**, left-aligned with the input’s left edge.
  - Helper text uses secondary text color; error text uses caption size with the destructive color.

- **Vertical rhythm**
  - **16px spacing between fields** in the same logical group.
  - **24px spacing between groups/sections** within a form.
  - This keeps generated forms visually aligned with the rest of the spacing system.

#### When Fields Should Span “Full Width”

- **Full-width fields are appropriate when:**
  - The content can be arbitrarily long (URLs, queries, JSON/YAML, long descriptions).
  - The form is a primary task on the page (new alert, new data source, new pipeline) and the field is the main focus of that step.
  - The form is inside a constrained container (drawer, narrow panel), where “full width” is already limited by the container.

- **Even then, “full width” means full form column, not full viewport**
  - The rule for primary-task forms is: **field width = 100% of the form column**, and the **form column itself has a max-width**.
  - This avoids stretched, edge-to-edge inputs on ultrawide screens while keeping layouts responsive down to tablet sizes.

## Principle 5: Navigation Legibility — Wayfinding in a Deep Product

### The Principle

At every point in the application, the user must be able to answer three questions without effort: Where am I? How did I get here? Where can I go?

### What This Means

vuSmartMaps has deep navigation hierarchies: Platform > Data Ingestion > ContextStreams > [Specific Pipeline] > [Plugin Configuration]. If the user loses context at any level, they lose confidence. Wayfinding is the system of cues — breadcrumbs, sidebar state, page titles, URL structure — that keeps users oriented.

### Why This Matters for vuSmartMaps

The pattern audit identified that the current product uses a numbered hierarchical sidebar with many nested levels. As the product adds modules (and it will — the SaaS transition will bring multi-tenancy, marketplace, and more), this hierarchy deepens. Without strong wayfinding, users will rely on browser back buttons and muscle memory, both of which break under change.

The command palette gap (Section 4.1 of the audit) and global search gap (Section 4.5) both relate to this principle. Users need multiple navigation strategies: structured (sidebar), spatial (breadcrumbs), and search-based (command palette).

### Rules

1. **Breadcrumbs are mandatory on every page except the top-level landing/home page.** Format: `Module > Section > Entity Name`. Breadcrumb segments are clickable links to their respective pages. The final segment (current page) is not a link and is visually distinguished (bolder weight or different color).

2. **The sidebar must visually indicate the current location.** The active item in the sidebar tree is highlighted (background color change). Its parent nodes are expanded. Other branches may be collapsed. When the user navigates to a page via deep link, breadcrumb, or search, the sidebar must sync to reflect the new location.

3. **URL structure must mirror the navigation hierarchy.** URLs are human-readable and hierarchical:
   - `/alerts` → Alert console (list)
   - `/alerts/:id` → Alert detail
   - `/alerts/:id/edit` → Alert edit
   - `/settings/data-sources` → Data source settings
   - `/settings/data-sources/:id` → Specific data source config

   This is not a cosmetic concern. Meaningful URLs support bookmarking, sharing, and AI-tool navigation. A URL like `/page?id=a3f2&tab=2` fails at all three.

4. **Command palette (Cmd+K / Ctrl+K) must be available globally.** It searches across:
   - Pages (navigate to "Alert Console," "Data Sources," etc.)
   - Entities (find a specific alert, dashboard, pipeline by name or ID)
   - Actions (create alert, export report, etc.)
   Results appear as the user types, categorized by type. Keyboard navigation (arrow keys + enter) selects and executes.

5. **Deep linking must preserve context.** If a user clicks a link in an alert notification that says "Alert #4521 fired on prod-db-03," the application must open to the alert detail page for #4521, pre-scrolled to the relevant section, with the sidebar reflecting the correct location. The user should be action-ready within one second of the page loading.

6. **Back navigation must work predictably.** The browser back button must always take the user to the previous page they were on within the application. Drawer opens and closes should not create history entries (opening a drawer is not a navigation event). Tab switches within a page should not create history entries. Only full page changes create history entries.

### Anti-Patterns

- Breadcrumbs that show generic labels ("Home > Page > Detail") instead of actual entity names ("Alerts > Alert #4521 > Edit").
- Opening a new page without updating the URL. If the user can't bookmark it, it doesn't exist as a page.
- Sidebar navigation that doesn't sync with direct URL navigation (user navigates via URL, but sidebar still shows the previous location).
- Command palette that only searches page names but not entity names. Users don't think in terms of "I want the Alert Console"; they think "I want alert #4521."

## Principle 6: Data Presentation — Tables, Charts, and Metrics

### The Principle

Data visualization exists to accelerate pattern recognition. Every visual encoding (position, color, size, shape) must map to a meaningful data dimension. If a visual element does not help the user find patterns or anomalies faster, remove it.

### What This Means

Most time in vuSmartMaps is spent looking at data: tables of alerts, time-series charts, metric summaries, log streams. The design system's data presentation components are the most-used and most-impactful components. Getting them right has disproportionate returns.

### Why This Matters for vuSmartMaps

The product's Grafana heritage gives it strong chart capabilities. The gap is in the surrounding structure: how tables, metrics, and charts compose into coherent pages, how they cross-reference, and how they respond to filtering and time range changes.

### Rules

#### Tables

1. **Column alignment:** Left-align text. Right-align numbers. Left-align dates (they're read as text patterns, not compared numerically). Center-align status icons and checkboxes.

2. **Row striping: Do not use alternating row colors (zebra striping).** Instead, use subtle horizontal dividers (1px border at 0.06 opacity) and adequate row height (36–40px). Zebra striping adds visual noise that competes with severity colors and selection highlights. If the table has many rows and scanability is a concern, show a hover highlight (subtle background change) on the row under the cursor.

3. **Column headers are frozen (sticky) when the table scrolls vertically.** The user must always be able to see what column they're reading.

4. **Default sort order must be meaningful for the entity type.** Alerts default to severity (critical first), then creation time (newest first). Logs default to timestamp (newest first). Reports default to last-modified time. Never leave default sort as arbitrary or alphabetical by first column.

5. **Cell content overflow: truncate with ellipsis and show full content in a tooltip on hover.** For long text fields (alert summaries, descriptions), truncate at 2 lines with "..." and a tooltip. Never wrap cell content onto multiple lines for columns that contain variable-length text — it creates unpredictable row heights that break scanning.

6. **Empty cells: show a dash "—" (em-dash), not blank space.** A blank cell is ambiguous (missing data? null? failed to load?). An em-dash explicitly means "no value."

7. **Bulk selection: checkboxes in the first column.** "Select all" checkbox in the header selects all visible rows. If filtered, show "N of M selected." Bulk action toolbar appears when at least one row is selected, either as a sticky bar above the table or replacing the filter toolbar.

#### Charts and Visualizations

8. **Time-series charts: x-axis is always time, right-to-left (past on left, present on right).** Y-axis scale auto-adjusts to data range with 10–20% headroom above the max value. Axis labels are concise: "14:30" not "2026-03-13T14:30:00.000Z."

9. **Chart tooltips appear on hover over any data point.** They show the exact value, timestamp, and series name. If multiple series intersect, the tooltip shows all values at that timestamp, sorted by value (descending).

10. **Chart colors follow a fixed palette sequence** for multi-series charts. The palette must be distinguishable in both light and dark mode and by users with color vision deficiency. Use 8 distinct hues, then cycle with different lightness. Never rely on color alone — use line patterns (solid, dashed, dotted) or marker shapes as secondary differentiators when more than 4 series are displayed.

11. **Dashboard panel grids use a 12-column layout.** Panels snap to column boundaries. Minimum panel width: 3 columns (25% of content area). This ensures panels align predictably across all dashboards.

#### Metrics and KPIs

12. **Metric cards show: value, label, and trend indicator.** The value is the largest text element (24–32px, monospace or tabular-nums font feature). The label is below or above (14px, secondary color). The trend indicator is a small sparkline or a delta value with an up/down arrow.

13. **Large numbers are formatted with locale-aware grouping:** 1,247 not 1247. Percentages show one decimal: 99.7% not 99.73218%. Durations use human-readable units: "2h 15m" not "8100s" not "135 min."

14. **Numbers that represent severity or status are color-coded** using the severity palette. A "Critical: 5" metric card has the number in red. Do not color the background of the entire card — color the value text. Background color is reserved for selected/active states.

### Anti-Patterns

- Chart backgrounds with gridlines so prominent they compete with the data. Gridlines should be barely visible (0.08–0.1 opacity).
- Pie charts for comparing more than 5 categories. Use a horizontal bar chart instead.
- Using different chart types for the same data dimension on the same page (a bar chart for latency on one panel and a line chart for latency on another).
- Metric cards that show a number with no context. "42" means nothing. "42 active alerts" means everything.

## Principle 7: Motion and Animation — Movement as Language

### The Principle

Animation is a communication tool, not a decoration. Every animation must answer at least one of three questions: What changed? Where did it come from? What's happening now?

### What This Means

Micro-animations are a language. A spinner says "waiting." A slide-in says "this content came from over there." A color fade says "this value just changed." A bounce says "pay attention here." This language must be consistent — the same motion must always mean the same thing — and must be fast enough that it informs without delaying.

### Why This Matters for vuSmartMaps

The research identified several interaction patterns that involve state transitions: alert status changes (active → acknowledged → cleared), pipeline runs (idle → running → success/failure), bulk operations (selected → processing → complete), and real-time dashboard updates (old value → new value). Without animation, these transitions are invisible — values just change between renders, and users miss them.

At the same time, vuSmartMaps users are often monitoring dashboards on secondary screens or scanning tables rapidly. Animations that are too slow, too prominent, or too frequent will be distracting rather than informative.

### Rules

1. **Transition duration scale:**
   - **Instant (0ms):** Checkbox toggles, radio selections, text input. Anything the user directly manipulates should respond with no visible delay.
   - **Swift (100–150ms):** Hover state changes (background color, border color), focus rings appearing, tooltip fade-in, dropdown opening. These are micro-feedback animations. Use `ease-out`.
   - **Standard (200–250ms):** Drawer slide-in/out, modal fade+scale, accordion expand/collapse, tab content crossfade. These are layout-level transitions. Use `ease-in-out`.
   - **Measured (300–400ms):** Page-level transitions (if any), large panel rearrangements, sidebar expand/collapse. Use `ease-in-out`. This is the maximum duration for any UI animation.

   Nothing in the product should animate for more than 400ms. If it takes longer than 400ms, it should be instant (skip the animation).

2. **Value change animation for live-updating numbers:** When a metric value changes on a dashboard refresh, briefly flash the background of the value cell with a subtle highlight (e.g., 150ms yellow-tinted flash that fades back to transparent over 300ms). This draws the eye to what changed without interrupting scanning. Do not animate the number counting up/down — just flash to indicate "this changed."

3. **Status transition animation:** When an entity changes state (alert severity changes, pipeline status changes), apply a brief pulse animation to the status badge: the badge scales to 1.05x and back over 200ms. This is subtle enough to catch peripheral vision without being distracting.

4. **Loading and progress animations:**
   - Skeleton screens use a shimmer animation (a gradient sweep from left to right over 1.5s, repeating). The shimmer moves at a consistent speed across all skeleton elements on the page.
   - Indeterminate progress bars use a left-to-right fill animation that repeats every 2s.
   - Spinner rotation: 1 full rotation per second. Use a simple ring spinner, not an elaborate multi-element loading animation.

5. **Drawer and modal entry/exit:**
   - Drawers slide in from the right edge over 250ms (ease-out on enter, ease-in on exit).
   - Modals fade in (opacity 0→1) and scale up (0.95→1.0) simultaneously over 200ms.
   - Backdrop dims over the same duration as the overlay element.

6. **No animation on initial page load.** When a page first renders, content appears instantly. No staggered fade-ins, no slide-up effects, no orchestrated entrance sequences. The user navigated here to see content, not to watch it arrive.

7. **Respect `prefers-reduced-motion`.** When the user's OS-level setting requests reduced motion, all animations except loading spinners and progress bars should be replaced with instant state changes. Loading indicators remain animated because they communicate essential state (something is happening).

### Anti-Patterns

- Staggered list item animations where each row fades in 50ms after the previous one. This delays content visibility for no informational gain.
- Bounce effects on buttons. A button should feel like pressing a physical switch: instant tactile feedback (active state change on press), not a spring.
- Parallax scrolling or scroll-triggered animations anywhere in the product. This is not a marketing site.
- Animating sidebar navigation open/close during initial page load (decide the state server-side based on user preference and render it).
- Spinning or animating logos/icons in the AppShell.

## Principle 8: Color as Semantic Signal

### The Principle

Color carries meaning. Every color in the palette maps to a specific semantic role. No color is decorative. If you cannot name what a color means, it should not be in the interface.

### The Model

The color system is **semantic and parametric**, not a bag of hex codes:

- **Semantic tokens first.** We define colors as roles — `primary`, `secondary`, `accent`, `muted`, `error`, `warning`, `success`, `info`, and their `*-foreground` counterparts. Components only reference these tokens (e.g., `--primary`, `--error-foreground`), never raw hex values.
- **Each role has a tiny HSL ramp.** For each role we keep a small, fixed set of uses: soft background, solid background, border/ring, foreground (text/icons), and ghost/hover. All of these are derived from a shared \(H, S\) pair for that role, with different \(L\) (lightness) values and, where necessary, opacity. Changing a role’s hue or saturation updates all of its uses consistently.
- **States are offsets, not new colors.** Default, hover, active, and disabled states are defined as predictable adjustments to the base role (e.g., hover = base \(L\) ± a fixed step, disabled = reduced saturation + higher lightness toward the surface background). No state introduces a brand‑new, unrelated color.
- **Brand vs. severity are separate systems.** Brand accent colors power primary actions, navigation, and focus rings. Severity colors (critical, error, warning, info, success, unknown) are a separate, fixed ramp and are reserved purely for status/health signalling.
- **Implementation detail lives in tokens.** When the marketing brand guide changes hues or introduces new “official” reds, ambers, or blues, we only update the underlying HSL/OKLCH values and lightness steps inside the token definitions. Component code and layouts do not change.

This ensures any future palette refresh (from brand or accessibility work) becomes a token update exercise, not a ground‑up recoloring of the product.

### What This Means

In an observability platform, color is infrastructure. Users don't just see colors — they read them. Red means something is broken. Yellow means something needs attention. Green means things are healthy. These associations are pre-trained by years of exposure to monitoring tools, traffic lights, and warning systems. vuSmartMaps must not violate these associations.

Beyond severity, color encodes categories (which service, which team, which data source), interactive states (hover, active, focused, disabled), and structural roles (background, surface, border, text).

### Rules

1. **Severity palette (fixed, non-overridable):**
   | Severity     | Light Mode          | Dark Mode           | Usage                              |
   |-------------|---------------------|---------------------|------------------------------------|
   | Critical    | `red-600` (#dc2626) | `red-500` (#ef4444) | Active critical alerts, system-down indicators |
   | Error/High  | `orange-600` (#ea580c) | `orange-400` (#fb923c) | High-severity alerts, errors, failures |
   | Warning     | `amber-600` (#d97706) | `amber-400` (#fbbf24) | Warnings, thresholds approaching  |
   | Info        | `blue-600` (#2563eb) | `blue-400` (#60a5fa) | Informational alerts, notices     |
   | Success/Healthy | `green-600` (#16a34a) | `green-400` (#4ade80) | Healthy systems, successful operations |
   | Unknown/Neutral | `gray-500` (#6b7280) | `gray-400` (#9ca3af) | Unknown state, pending, N/A       |

   These values are chosen for WCAG AA contrast compliance against both light and dark backgrounds. The light mode values use the 600 weight (darker) because they sit on white/light backgrounds. Dark mode values use the 400–500 weight (lighter) because they sit on dark backgrounds.

2. **Severity colors are reserved exclusively for severity.** Do not use red for a "Cancel" button. Do not use green for a "Next" button. Do not use amber for a highlight. These colors belong to the severity system. Buttons and interactive elements use the brand accent color and neutral gray.

3. **Interactive state colors:**
   | State      | Treatment                                       |
   |------------|--------------------------------------------------|
   | Default    | Component's base color                           |
   | Hover      | Slight background tint change (opacity shift, e.g., `bg-primary/90`) |
   | Active/Pressed | Darker shade of hover state                  |
   | Focused    | 2px ring in accent color with 2px offset from element edge |
   | Disabled   | 40% opacity of default state. No hover/active changes. Cursor: `not-allowed` |
   | Selected   | Accent background tint (10% opacity of accent color) with accent left border (3px) |

4. **Background layering (light mode):**
   - **Base background:** White (`#ffffff`) — page background
   - **Surface:** Subtle warm gray (`#fafaf9` or similar) — cards, panels, sidebar, elevated surfaces
   - **Inset:** Slightly darker gray (`#f4f4f5`) — input fields, code blocks, table headers
   - **Border:** `#e4e4e7` — all structural borders

   In dark mode, this inverts to a layered dark scale: base → surface → inset using progressively lighter grays.

5. **Data visualization palette (for charts):** A fixed sequence of 8 hues chosen for mutual distinguishability, including for deuteranopia and protanopia:
   - Blue, Orange, Teal, Pink, Purple, Yellow-green, Coral, Cyan
   - These are NOT the severity colors. Chart series colors and severity colors are independent systems.

6. **Text colors have exactly three levels:**
   - **Primary text:** Highest contrast. Used for headings, values, active labels. (`gray-900` light / `gray-100` dark)
   - **Secondary text:** Reduced contrast. Used for descriptions, supporting labels, timestamps. (`gray-600` light / `gray-400` dark)
   - **Tertiary text:** Lowest contrast. Used for placeholders, disabled labels, de-emphasized metadata. (`gray-400` light / `gray-600` dark)

   No other text color variations. If you need a fourth level, you're over-complicating the hierarchy.

7. **The brand accent color is used sparingly.** It appears on: primary action buttons, active navigation items, links, focus rings, and selected-state indicators. It does NOT appear on backgrounds, large areas, or decorative elements. The accent is a signal, not a theme.

### Anti-Patterns

- Using red for UI elements that are not severity-related (red "Cancel" buttons, red required-field indicators). Use the neutral destructive variant for cancel/delete and an asterisk or text label for required fields.
- Using blue for both informational severity AND links. Links should use the brand accent color, which must be distinct from the Info severity blue. If the brand accent is close to blue, shift the Info severity toward a cerulean or sky blue to maintain distinction.
- Applying severity colors to chart backgrounds. A chart showing error rates should use the standard chart color palette, not red fills. The severity reading comes from the VALUE, not the chart color.
- Using low-contrast text in dark mode. Dark mode is not an excuse for gray-on-gray. The contrast ratios must meet WCAG AA (4.5:1 for body text, 3:1 for large text) in both modes.

## Principle 9: Typography as Hierarchy Machine

### The Principle

The type system creates the primary visual hierarchy on every screen. Type size, weight, and family are the three levers. Use them deliberately and consistently. If you need a fourth lever (color, decoration, case), the first three are not being used correctly.

### What This Means

In data-dense interfaces, layout tricks (cards, borders, backgrounds) get crowded fast. Typography remains the most reliable way to establish what's important, what's secondary, and what's supporting. A well-built type scale does more organizational work than any amount of boxing and bordering.

### Rules

1. **Two font families, three roles:**
   - **Sans-serif (IBM Plex Sans or equivalent):** All UI text — headings, body, labels, buttons, navigation.
   - **Monospace (IBM Plex Mono or equivalent):** All data values that need alignment (numbers in tables, timestamps, code, IDs, IP addresses, JSON/YAML), all code/config editors.

   No other font families. No display fonts, no serif fonts, no handwriting fonts.

2. **Type scale (relative to the 14px data-context base):**
   | Token        | Size   | Weight   | Line Height | Usage                                    |
   |-------------|--------|----------|-------------|------------------------------------------|
   | `display`   | 28–32px | 600      | 1.2         | Page-level titles only                   |
   | `heading-1` | 22–24px | 600      | 1.25        | Section headings                         |
   | `heading-2` | 18–20px | 600      | 1.3         | Sub-section headings, card titles        |
   | `heading-3` | 16px   | 600      | 1.35        | Group labels, field group headings       |
   | `body`      | 14px   | 400      | 1.5         | Default text, descriptions, table cells  |
   | `body-sm`   | 13px   | 400      | 1.45        | Secondary info, timestamps, metadata     |
   | `caption`   | 12px   | 400      | 1.4         | Tertiary text, helper text, footnotes    |
   | `code`      | 13px   | 400 (mono) | 1.5       | Inline code, IDs, values in tables       |
   | `code-block` | 13px  | 400 (mono) | 1.6       | Code editors, JSON/YAML views, log lines |

3. **Weight usage is binary for most contexts: 400 (regular) and 600 (semibold).** Do not use 300 (light) — it's too thin at small sizes on low-DPI screens. Do not use 700+ (bold/extrabold) except for the `display` token if a heavier presence is needed. Semibold (600) is the "emphasis" weight; it's visible without being loud.

4. **Letter-spacing:** Negative letter-spacing (-0.01em to -0.02em) on `display` and `heading-1` only. All other sizes use default (0) letter-spacing. Tight letter-spacing at small sizes hurts readability.

5. **Tabular numbers for all numeric data in tables.** Use the `font-variant-numeric: tabular-nums` CSS property (or the Tailwind `tabular-nums` utility) on all table cells that contain numbers. This ensures digits align vertically across rows, which is critical for comparing values in a column.

6. **No uppercase/all-caps for labels or headings.** Uppercase text is harder to scan (letter shapes are less differentiated in capitals). If you need to de-emphasize a label while maintaining visibility, use `caption` size with secondary text color. The one exception: short status badges (e.g., "CRITICAL", "ACTIVE") may use uppercase at 11px with increased letter-spacing (+0.05em) as a conventional pattern, because users expect badge text to be capitalized in this domain.

7. **Truncation rules:** Single-line truncation with ellipsis for table cells and list items. Multi-line truncation (2 lines with ellipsis) for descriptions and summaries in cards. No truncation for headings — if a heading doesn't fit, the layout needs to accommodate it, not truncate it.

### Anti-Patterns

- Using font size to create emphasis within a paragraph. If a word needs emphasis, use semibold weight, not a larger size. Size changes mid-paragraph break reading rhythm.
- Using more than two weights on a single screen. If you find yourself reaching for light (300) and bold (700) alongside regular and semibold, the hierarchy is over-complicated.
- Setting line-height below 1.3 for any multi-line text. Single-line elements (buttons, badges, single-line table cells) can use tighter line-heights, but anything that wraps needs at least 1.3.
- Using monospace font for things that are not code or data values. Labels, descriptions, and navigation items are not code.

## Principle 10: Error and Empty States — Failure as a Design Opportunity

### The Principle

Every screen has a failure mode and a zero-data mode. Both must be designed explicitly, not left as afterthoughts. Error states guide recovery. Empty states guide first action.

### What This Means

The pattern audit identified empty states as a gap in the current product (Section 4.3 of the audit). In a SaaS model, every new tenant starts with empty modules. Every new feature starts with zero data. The empty state is the first impression for each module. If it's a blank white screen with no guidance, the user's reaction is "Is this broken?" If it's a purposeful screen with a clear next step, the reaction is "I know what to do."

Similarly, errors will happen. APIs will fail. WebSocket connections will drop. Permission checks will deny access. The error state is a trust moment: handle it well and the user's confidence increases; handle it poorly and they lose faith in the product.

### Rules

1. **Empty state anatomy:**
   ```
   ┌──────────────────────────────────┐
   │                                  │
   │       [Illustration / Icon]      │  (optional, 64–120px, muted color)
   │                                  │
   │        No [entities] yet         │  (heading-2, primary text color)
   │                                  │
   │   [1–2 sentence explanation of   │  (body, secondary text color)
   │    what this section is for and  │
   │    what having data here means]  │
   │                                  │
   │       [ + Create First One ]     │  (primary action button)
   │                                  │
   └──────────────────────────────────┘
   ```

   The heading names the missing entity. The description explains the value of the feature. The CTA is the single most common next action. Do not show multiple CTAs in empty states — decision paralysis at the zero state is fatal for onboarding.

2. **Error state anatomy:**
   ```
   ┌──────────────────────────────────┐
   │                                  │
   │        [Warning Icon]            │  (24–32px, severity-appropriate color)
   │                                  │
   │    Unable to load [entities]     │  (heading-2, primary text color)
   │                                  │
   │   [What went wrong, in plain     │  (body, secondary text color)
   │    language. Not a stack trace.   │
   │    Not "An error occurred."]     │
   │                                  │
   │         [ Retry ]                │  (primary button if retryable)
   │     [ View Details ]             │  (text link to technical details,
   │                                  │   collapsed by default)
   └──────────────────────────────────┘
   ```

   The heading says what failed. The description says why in human terms. The primary action is retry if the error is transient, or a corrective action if the user can fix it (e.g., "Check your permissions" with a link to settings). Technical details (error codes, request IDs, stack traces) are available but hidden behind "View Details" for debugging.

3. **Partial failure states:** When a page has multiple data-fetching regions and one fails, only the failed region shows an error state. The rest of the page renders normally. Do not blank the entire page because one API call failed. Show the error inline, within the component that failed, with a retry option specific to that component.

4. **Permission-denied states:** When a user navigates to a page they don't have access to, show a clear "You don't have access" message with:
   - What permission is needed ("This page requires the Alert Admin role")
   - Who can grant it ("Contact your organization administrator")
   - A "Go Back" button

   Do not redirect silently. Do not show a generic 404. The user needs to know the page exists but is gated.

5. **Error messages follow a three-part structure:**
   - **What happened:** "Failed to save alert configuration."
   - **Why it happened (if known):** "The evaluation query references a data source that no longer exists."
   - **What to do next:** "Update the query to use an active data source, or contact your admin to restore the deleted source."

   If the cause is unknown, say so: "An unexpected error occurred. If this persists, contact support with reference ID: abc-123."

6. **No error message may contain only a status code or technical identifier.** "Error 500" is not an error message. "Something went wrong" is not an error message. Both fail the three-part structure above.

### Anti-Patterns

- Blank screens with no text for empty states. Even a single line of "No alerts configured" is infinitely better than whitespace.
- Generic error messages across the entire product ("An error occurred. Please try again later."). The user doesn't know what errored, what "later" means, or whether trying again will help.
- Showing error toasts for non-transient failures. If a page fails to load, an error toast that disappears after 5 seconds leaves the user staring at a blank page with no explanation. Persistent inline errors for persistent failures; toasts for transient feedback only.
- Treating 404 and 403 the same way. A 404 means "this doesn't exist." A 403 means "this exists but you can't see it." Different root causes, different guidance.

## Principle 11: Configuration Safety — Editing with Confidence

### The Principle

Any action that modifies a production system's behavior must be protected by a proportional safety mechanism. The more consequential the change, the more explicit the confirmation.

### What This Means

vuSmartMaps users configure alert rules that page engineers at 3am, data pipelines that transform production telemetry, and notification routes that send messages to Slack channels with thousands of people. A misconfiguration is not a cosmetic problem — it's an operational incident. The UI must make it difficult to make consequential changes accidentally, while keeping non-consequential changes frictionless.

### Why This Matters for vuSmartMaps

The pattern audit identified "Diff / Review Changes Before Apply" as a gap (Section 4.2). The current product applies configuration changes directly on save. As the product moves to multi-tenant SaaS, the blast radius of misconfiguration grows. A safety-conscious editing experience is a competitive differentiator for an observability platform.

### Rules

1. **Safety tiers based on consequence:**

   | Tier | Consequence | Safety Mechanism | Examples |
   |------|------------|-----------------|----------|
   | **Low** | User-local, easily reversible | Save with success toast | Dashboard layout change, personal filter preset |
   | **Medium** | Affects team/tenant, reversible | Save with "Unsaved changes" indicator + confirmation dialog | Alert rule edit, notification route update |
   | **High** | Affects production behavior, hard to reverse | Diff/review before apply + explicit "Apply Changes" step | Pipeline config, data source credentials, RBAC changes |
   | **Destructive** | Irreversible data loss | Confirmation dialog requiring typed entity name | Delete data source, purge logs, remove user |

2. **Diff/review for Tier High changes:**
   - Show a side-by-side or unified diff of the before/after state before the user commits.
   - Summary at the top: "3 fields changed, 1 section added, 0 sections removed."
   - "Apply Changes" button is the only way to commit. No auto-save for high-tier changes.
   - Optional: "Test / Dry Run" button that validates the config against the live system without applying it (analogous to Terraform plan).

3. **Destructive action confirmation dialog:**
   - Title: "Delete [entity name]?"
   - Body: Specific consequences. "This will permanently delete the pipeline 'prod-metrics-agg' and stop all data processing through it. This action cannot be undone."
   - Confirmation input: The user must type the entity's name to enable the delete button. This is the strongest confirmation pattern and should be reserved for truly irreversible actions.
   - The "Delete" button is styled as destructive (not red background — use a red text label with an outlined/ghost button style, so the button doesn't attract accidental clicks).

4. **Undo for reversible actions.** When a user performs a medium-consequence action that is reversible (archive an alert, disable a notification route), show an undo snackbar at the bottom of the screen: "[Entity] disabled. [Undo]" The snackbar persists for 8 seconds, then the action is committed. This is faster than a confirmation dialog and provides a better experience for users who are confident in their actions.

5. **Form validation is real-time and inline.** Validation errors appear immediately below the offending field as the user types (with a debounce of 300ms for text inputs, immediate for selects/toggles). Do not wait until form submission to show validation errors. The submit button may be disabled while validation errors exist, but it must still be visible (not hidden) — a disabled button with a tooltip explaining "Fix 2 errors above" is more helpful than a button that appears only when the form is valid.

6. **Auto-save is only for Tier Low changes.** Dashboard layout changes, personal preferences, and view customizations may auto-save. Configuration that affects system behavior never auto-saves. The user must explicitly click "Save" or "Apply."

### Anti-Patterns

- A single "Are you sure?" dialog for all destructive actions regardless of consequence. Deleting a dashboard filter preset and deleting a data source are not the same risk level.
- Save buttons that are always enabled and show an error after clicking. Validate before submission.
- No visual diff before applying config changes. Forcing users to remember what they changed is asking for mistakes.
- Auto-saving pipeline configurations. A partially-edited pipeline that auto-saves in an intermediate state can break data processing.

## Principle 12: Responsive Behavior — Adapting Without Losing Function

### The Principle

Layouts adapt to viewport size, but no feature becomes inaccessible at any supported viewport. The product is primarily designed for large screens (1440px+) and remains fully functional down to 1024px. Below 1024px, data-heavy features degrade gracefully but may reduce density.

### What This Means

vuSmartMaps users primarily work on desktop monitors and laptops. There is no native mobile use case for pipeline configuration or log analysis. However, dashboard viewing and alert triage may happen on tablets, and some stakeholders check dashboards on smaller laptops. The responsive strategy is: desktop-first, functional down to tablet, legible (but not full-featured) below that.

### Rules

1. **Breakpoint scale:**
   | Breakpoint | Width      | Adaptation                                   |
   |-----------|------------|----------------------------------------------|
   | Desktop XL | ≥1440px   | Full layout. Wide tables. Side-by-side panels. |
   | Desktop    | 1280–1439px | Full layout. Slightly narrower tables.       |
   | Laptop     | 1024–1279px | Sidebar collapses to icon rail by default. Tables may hide low-priority columns. |
   | Tablet     | 768–1023px  | Sidebar becomes hamburger overlay. Drawers become full-width sheets. Split panes stack vertically. |
   | Mobile     | <768px     | Not a primary target. Show read-only dashboard views. Configuration and editing features show a "Use a larger screen for full functionality" message. |

2. **Tables on narrow viewports:** Below 1280px, tables automatically hide columns beyond the first 5–6 (user-configurable via column picker). A "+" icon at the end of each row expands to show hidden column values in a stacked key-value layout below the row.

3. **Sidebar behavior:**
   - ≥1280px: Expanded (240px) by default. User can collapse to icon rail (48px). Preference persists.
   - 1024–1279px: Icon rail by default. User can expand temporarily (overlay, not pushing content).
   - <1024px: Hidden by default. Hamburger icon in the top bar triggers a full-height overlay sidebar.

4. **Drawers on small viewports:** Below 1024px, right-side drawers become full-width sliding sheets (from the bottom on mobile, from the right on tablet but taking 100% width). The dismiss gesture on touch devices is a downward swipe.

5. **Dashboard panels on narrow viewports:** The 12-column grid compresses. Below 1024px, all panels become full-width (12 columns), stacked vertically. This loses the side-by-side comparison capability, but maintains readability.

6. **No horizontal scrolling on the page level.** If a table is too wide, the table scrolls horizontally within its container, but the page itself does not scroll horizontally. The sidebar and top bar remain fixed.

### Anti-Patterns

- Hiding critical actions behind responsive breakpoints. The "Create Alert" button must be accessible at every viewport, even if it moves from the page header to a floating action button on mobile.
- Responsive font size scaling that changes the type hierarchy. The type scale is fixed. The layout adapts around it.
- Removing chart interactivity (tooltips, zoom) on touch devices. Touch users still need to inspect data points.

## Principle 13: Accessibility as a Constraint

### The Principle

Every component is keyboard-navigable, screen-reader-compatible, and meets WCAG 2.1 AA contrast requirements. Accessibility is not a checklist applied after design; it is a constraint that shapes design from the start.

### What This Means

Accessible patterns are better patterns. Focus management that works for screen readers also works for keyboard power users. Color contrast requirements that satisfy WCAG also improve readability for users in bright-lit SOC rooms. Touch targets sized for motor impairment also work better with trackpads. Designing within accessibility constraints produces better UX for everyone.

### Rules

1. **Keyboard navigation:** Every interactive element is reachable via Tab/Shift+Tab. Every operable element is activatable via Enter or Space. Dropdown menus, tree views, and tab bars support arrow key navigation. Modals and drawers trap focus while open and return focus to the trigger element on close. Escape closes the topmost overlay.

2. **Focus indicators:** All focusable elements show a visible focus ring. The ring is 2px wide, in the accent color, with a 2px offset from the element edge. The focus ring is visible in both light and dark mode. Never remove focus outlines without replacing them with an equally visible alternative.

3. **ARIA landmarks and roles:**
   - The sidebar navigation uses `<nav>` with `aria-label="Main navigation"`.
   - The main content area uses `<main>`.
   - The top bar uses `<header>`.
   - Modals use `role="dialog"` with `aria-modal="true"` and `aria-labelledby` pointing to the dialog title.
   - Drawers use `role="complementary"` or `role="dialog"` depending on whether they're informational or action-oriented.
   - Tables use proper `<table>`, `<thead>`, `<th>`, `<tbody>`, `<tr>`, `<td>` elements, not div-based layouts with ARIA roles.

4. **Skip-to-main-content link:** The first focusable element on any page is a visually-hidden skip link that becomes visible on focus: "Skip to main content." It jumps focus to the `<main>` element.

5. **Color is never the sole indicator.** Any information conveyed by color (severity, status, chart series) must also be conveyed by a secondary channel: text label, icon, pattern, or position. A colorblind user must be able to distinguish critical from warning alerts without relying on red vs. amber. This means severity badges always include text labels ("Critical," "Warning"), not just colored dots.

6. **Contrast ratios:**
   - Body text: ≥4.5:1 against its background.
   - Large text (≥18px or ≥14px bold): ≥3:1.
   - UI components and graphical objects: ≥3:1 against adjacent colors.
   - These ratios apply in both light and dark mode.

7. **Touch targets:** All interactive elements have a minimum clickable/tappable area of 44×44px. For elements that are visually smaller (icon buttons, table row action icons), the clickable area extends via padding, even if the visual element is 24×24px.

8. **Live regions for dynamic content:** When data updates asynchronously (alert count changes, toast notifications appear, progress completes), use `aria-live="polite"` for non-urgent updates and `aria-live="assertive"` for critical ones. This ensures screen readers announce changes without requiring the user to navigate to the changed element.

### Anti-Patterns

- Using `div` with `onClick` instead of `button` for interactive elements. Divs are not keyboard-accessible by default and have no implicit ARIA role.
- Custom checkboxes and radio buttons that lose native keyboard behavior. Use native inputs with visual styling via CSS, or Radix UI primitives that preserve accessibility.
- Placeholder text as the only label for form fields. Placeholders disappear on input and are not reliably announced by screen readers. Every input must have a visible `<label>` (or an `aria-label` if visual space is truly constrained).
- Using `title` attributes as the primary way to convey information. Screen readers handle `title` inconsistently. Use `aria-label`, `aria-describedby`, or visible text.

## Principle 14: AI-Native Design System Architecture

### The Principle

The design system must be structured so that an AI tool (Cursor, Claude, Copilot, v0) can generate a complete, production-quality screen from a product requirement description, and that screen must be visually indistinguishable from one built by a human designer.

### What This Means

This is the core promise of vuDS-x. The design system is not just a component library for human developers — it is the knowledge base that AI tools use to generate correct, consistent interfaces. For this to work, the system must be predictable at every level: file structure, naming, prop interfaces, composition patterns, and visual output.

The AI-readiness research (from the primer document) established three tiers: static discovery (llms.txt, schemas), programmatic access (MCP), and structural predictability (uniform patterns). This principle focuses on the structural predictability tier, which is the most impactful.

### Rules

1. **Every component follows the same structural pattern:**
   - Variants defined via CVA (Class Variance Authority) or an equivalent single-source variant system.
   - Class merging via a single `cn()` utility.
   - Props extend native HTML element types via `React.ComponentProps<"element">`.
   - Named exports only (no default exports).
   - One component per file. File name matches component name in kebab-case.
   - `data-slot` attributes on all semantic sub-elements for CSS targeting.

   An LLM that has seen the Button component's source code must be able to predict the structure of any other component in the system.

2. **Page templates are composable from named layout primitives:**
   - `PageHeader` — title, breadcrumbs, actions
   - `PageToolbar` — search, filters, view toggles
   - `PageContent` — the main body area
   - `PageFooter` — pagination, sticky action bar

   An AI generating a "list page" imports and composes these primitives. It does not create ad-hoc layout divs. This ensures every generated page conforms to the spatial contract (Principle 4).

3. **Component prop interfaces are self-documenting via TypeScript.**
   - All props have JSDoc descriptions.
   - Required props are not optional. Optional props have sensible defaults.
   - Complex props use discriminated unions, not loose string types. A button variant is `"primary" | "secondary" | "ghost" | "destructive"`, not `string`.
   - Event handler props follow React conventions: `onAction`, `onChange`, `onDismiss`.

4. **Design tokens are CSS custom properties, not JS constants.** AI tools generating Tailwind classes or CSS can reference tokens directly without importing JavaScript modules. The token names are semantic (`--color-severity-critical`, `--spacing-page-gutter`) not abstract (`--color-red-600`, `--spacing-6`).

5. **The component registry is machine-readable.** A JSON (or JSON-compatible) registry lists every component with:
   - Name
   - Category (navigation, data-display, forms, feedback, overlay, layout)
   - File path
   - Dependencies (other vuDS-x components it uses)
   - Peer dependencies (external libraries)
   - Variants and their values
   - Usage examples (at least one per variant)

   This registry is what the MCP server queries. It is the single source of truth.

6. **Pattern templates are documented as composable recipes.** Each of the 18 layout templates from the research (Admin View, Analytics View, List View, etc.) is documented as a composition recipe: which page primitives, which components, in what arrangement, with what data flow. An AI receiving a prompt like "create an alert list page" should be able to look up the List View recipe and generate the correct composition.

### Anti-Patterns

- Components with inconsistent APIs. If Button uses `variant` but Badge uses `type` for the same concept (visual variant), AI will generate incorrect prop names half the time.
- Hardcoded styles in components instead of token references. An AI generating a new component will copy the hardcoded value instead of using the token, and the value will fall out of sync.
- Components that require complex setup (context providers, initialization hooks) to render. AI-generated screens should work with minimal boilerplate.
- Undocumented component behaviors. If a component has a non-obvious behavior (auto-focuses on mount, traps scroll, injects portal), it must be documented in the component's JSDoc and registry entry. AI cannot infer what it has not been told.

## Principle 15: Consistency Over Cleverness

### The Principle

When facing a design decision, choose the solution that most closely matches an existing pattern in the system, even if a novel solution would be marginally better for the specific case. Homogeneity across the product is more valuable than local optimization.

### What This Means

The core quality metric for vuSmartMaps is this: can a user navigate from the Alert Console to the Report Builder to the Pipeline Editor and feel like they're using the same product? If each module's design team (or each AI generation session) optimizes independently, the product becomes a collection of well-designed modules that feel disjointed together.

Consistency means using the same component for the same purpose, every time. The same table component with the same column behaviors. The same drawer width and animation. The same form layout and validation pattern. The same empty state structure. The same error handling.

### Rules

1. **Solved problems stay solved.** Once a pattern is established in the design system (e.g., "how we do filter bars"), every new screen that needs a filter bar uses that pattern. No reinvention. If the established pattern has a flaw, fix it at the system level (update the design system component), not at the screen level (build a one-off alternative).

2. **Deviations require a documented reason.** If a developer or AI needs to deviate from an established pattern, the deviation must be documented: what pattern was deviated from, why, and whether the deviation should become the new standard. This is a code review checkpoint, not a bureaucratic process — a one-line comment in the PR description is sufficient.

3. **New patterns start as proposals.** When a new interaction need arises that isn't covered by existing patterns (e.g., a visual pipeline editor), the first implementation becomes the reference. Document it immediately. Do not allow two different implementations of the same novel pattern to emerge in parallel.

4. **Component reuse over custom builds.** If the design system has a Select component, use it — even if a custom dropdown would look 10% better for this specific use case. The Select component is tested, accessible, documented, and understood by AI tools. The custom dropdown is none of those things.

5. **Cross-module visual alignment:** All list pages look the same. All detail pages look the same. All settings pages look the same. The content within them varies by module, but the structural elements (header, toolbar, content area, pagination) are identical in position, spacing, and behavior.

### Anti-Patterns

- "This module is special, so we designed it differently." No module is special enough to justify breaking spatial consistency. If a module has unique needs, extend the system to accommodate them, don't work around it.
- Building one-off components for single-use cases that could be served by composing existing components. A "severity summary bar" is not a new component — it's a horizontal stack of Badge components with counts.
- Using different animation timings or easing curves in different parts of the product. The motion language (Principle 7) is global.
- Generating screens with AI tools that have not been configured with the full design system context. A screen generated without knowledge of these principles will look "off" — not wrong enough to catch in a screenshot, but wrong enough that a user subconsciously feels the inconsistency.

## Summary: The Checklist

For every screen, page, or component — whether built by hand or generated by AI — verify:

| #  | Check | Principle |
|----|-------|-----------|
| 1  | Is information density appropriate? No wasted space, no visual noise? | P1 |
| 2  | Is complexity layered? Can a casual user and a power user both find their depth? | P2 |
| 3  | Does every data-dependent area handle loading, empty, error, populated, and refreshing states? | P3 |
| 4  | Does the page follow the standard anatomy (header → toolbar → content → footer)? | P4 |
| 5  | Can the user tell where they are via breadcrumbs, sidebar state, and URL? | P5 |
| 6  | Are tables, charts, and metrics following the data presentation rules? | P6 |
| 7  | Is every animation communicating something? Is anything slower than 400ms? | P7 |
| 8  | Are colors used semantically? Is severity reserved for severity? | P8 |
| 9  | Does the type hierarchy use only the defined scale and weights? | P9 |
| 10 | Do empty and error states guide the user toward a next action? | P10 |
| 11 | Are consequential changes protected by proportional safety mechanisms? | P11 |
| 12 | Does the layout work at 1440px, 1280px, and 1024px? | P12 |
| 13 | Is every interactive element keyboard-accessible with visible focus? | P13 |
| 14 | Does the component follow the structural pattern (CVA, cn, typed props, named export)? | P14 |
| 15 | Does this screen look like it belongs in the same product as every other screen? | P15 |

If any check fails, fix it before shipping.
